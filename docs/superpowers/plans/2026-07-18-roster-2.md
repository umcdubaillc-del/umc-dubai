# ROSTER-2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give each `wa_team` number three independent capability flags (`cap_lead_alerts`, `cap_approve`, `cap_watchdog`) so every team-directed WhatsApp stream is gated on its own flag, with `active` as the master gate.

**Architecture:** Add three columns via the existing idempotent `addMissingColumns` pattern. Replace the single all-active broadcast selector `getActiveWaTeam` with `getWaTeamByCap(env, cap)`; every send fan-out reads exactly one cap. Approval authority becomes the union of the `cap_approve` roster plus a free-text override (minus any deactivated number), and can never silently lock out. Admin roster editor gains per-row toggles; the Assistant card's decision-numbers field becomes a read-through of `cap_approve` plus an override.

**Tech Stack:** Cloudflare Worker (`src/admin.js`, `src/index.js`), D1 (`BILLING_DB`), plain-JS admin UI embedded in `admin.js`. Deploy via `git push` ‚Üí CI gate `python3 build_pages.py && npm run check`. Live verification through the admin API + owner's Chrome session. Repo tests are standalone node `.mjs` scripts.

**Spec:** `docs/superpowers/specs/2026-07-18-roster-2-design.md` (approved). Keep the diff lean ‚Äî three columns, one helper, ~20 call-sites, toggles.

```js
// ROSTER-2 pure helpers (unit-tested in tests/test-roster2.mjs).
// Which cap a lead-alert send reads: watchdog for escalations, else lead_alerts.
export function capForLeadAlerts(opts) {
  return opts && opts.escalation ? "cap_watchdog" : "cap_lead_alerts";
}
// Authorized-approver set = (cap_approve roster) ‚à™ (override numbers), minus any
// number that is a deactivated wa_team row. `overrideRaw` is the raw
// app_settings string; numbers are normalized with waMeNumber.
export function mergeAuthorizedNumbers(capApproveNums, overrideRaw, deactivatedNums) {
  const dead = new Set((deactivatedNums || []).map((n) => waMeNumber(n)).filter(Boolean));
  const set = new Set();
  for (const n of capApproveNums || []) { const x = waMeNumber(n); if (x && !dead.has(x)) set.add(x); }
  const raw = overrideRaw ? String(overrideRaw).trim() : "";
  if (raw) for (const p of raw.split(/[,\s]+/)) { const x = waMeNumber(p); if (x && !dead.has(x)) set.add(x); }
  return set;
}
```

Note: `waMeNumber` already exists and is in scope in `admin.js`. The test's `norm` stand-in is illustrative only; the assertions use already-normalized `9715‚Ä¶` digit strings so `waMeNumber` is a no-op on them.

Replace 5276-5289 with:

```js
async function getAuthorizedDecisionNumbers(env) {
  // Primary source: wa_team with active=1 AND cap_approve=1.
  const approvers = (await getWaTeamByCap(env, "cap_approve")).map((m) => m.phone);
  // Deactivated wa_team numbers must be excluded even if they appear in the override.
  let deactivated = [];
  try {
    const { results } = await env.BILLING_DB.prepare(
      `SELECT phone FROM wa_team WHERE active = 0`
    ).all();
    deactivated = (results || []).map((r) => r.phone);
  } catch (e) { /* table absent ‚Üí no exclusions */ }
  // Free-text override adds exceptional (non-roster) numbers.
  let overrideRaw = "";
  try {
    const r = await env.BILLING_DB.prepare(
      `SELECT value FROM app_settings WHERE key='assistant_decision_numbers'`
    ).first();
    overrideRaw = r && r.value ? String(r.value) : "";
  } catch (e) { /* setting absent */ }
  return mergeAuthorizedNumbers(approvers, overrideRaw, deactivated);
}
```

Note the semantic change: previously the override *replaced* the roster and the roster was only a fallback. Now it is a strict **union**, and an empty override yields exactly the `cap_approve` roster.
